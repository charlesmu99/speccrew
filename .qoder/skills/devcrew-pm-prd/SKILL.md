# 触发场景

- PM Agent 接收到用户需求描述
- 用户说"写一个 PRD"、"帮我整理需求"、"新功能需求"

# 工作流程

## 步骤一：需求澄清

向用户提问，确认以下内容（未明确的都需要询问）：

| 问题 | 目的 |
|------|------|
| 这个功能解决什么问题？ | 确认背景与动机 |
| 目标用户是谁？ | 确认用户范围 |
| 核心场景是什么？ | 识别主流程 |
| 有哪些不在范围内？ | 明确边界 |
| 有没有验收标准？ | 确认完成定义 |

**原则**：宁可多问，不可假设。

## 步骤二：加载业务知识

读取以下文件，了解现有系统现状：

```
.devcrew-workspace/knowledge/bizs/modules/modules.md
.devcrew-workspace/knowledge/bizs/flows/flows.md
```

确认事项：
- [ ] 需求是否与现有模块重叠？如有重叠，是扩展还是新建？
- [ ] 需求涉及哪些现有业务流程？
- [ ] 是否有相关行业规范需要参考？（如有，读取 `domain/standards/`）

## 步骤三：读取 PRD 模板

读取模板文件：
```
.qoder/skills/devcrew-pm-prd/templates/PRD-TEMPLATE.md
```

## 步骤四：撰写 PRD

按模板结构填写，要求：
- **背景与目标**：说明为什么做、成功标准是什么
- **用户故事**：`作为 [用户角色]，我希望 [做什么]，以便 [达到什么目的]`
- **功能需求**：按优先级分组（P0 核心 / P1 重要 / P2 可选）
- **非功能需求**：性能、安全、兼容性等
- **验收标准**：可量化、可验证的完成定义
- **边界说明**：明确 Not In Scope 的内容
- **假设与依赖**：前置条件、外部依赖

## 步骤五：任务粒度检查

PRD 完成后，检查每个用户故事的粒度是否符合"单次迭代可完成"原则：

**合适的粒度（一次迭代完成）**：
- 添加一个数据库字段并编写迁移
- 在现有页面中新增一个 UI 组件
- 修改一段已有的服务端逻辑
- 给列表页加一个筛选条件

**需要拆分的粒度（过大）**：
- 实现一个完整的仪表板
- 新增一整套认证/登录体系
- 对现有 API 做整体重构
- 涉及多个模块的跨领域变更

**拆分策略**：
- 将大用户故事拆分为多个小用户故事
- 每个小用户故事有独立的验收标准
- 标注用户故事之间的依赖关系

如有需要拆分的故事，更新 PRD 后再进入下一步。

## 步骤六：确定存放路径

询问用户当前迭代编号，确定文件路径：
```
.devcrew-workspace/projects/p[编号]-[迭代名]/01.prds/[功能名]-prd.md
```

如迭代目录不存在，参考 `p000-sample` 目录结构创建。

## 步骤七：写入文件并请求确认

将 PRD 写入文件后，向用户展示摘要并明确请求确认：

```
PRD 已生成：.devcrew-workspace/projects/pXXX/01.prds/[功能名]-prd.md

请确认以下关键点：
1. 功能边界是否准确？
2. 验收标准是否可量化？
3. Not In Scope 是否完整？

确认无误后，可启动 Solution Agent 进行方案规划。
```

# 检查清单

- [ ] 所有不明确的需求已向用户澄清
- [ ] 已读取业务模块清单，确认与现有功能的边界
- [ ] PRD 按模板结构完整填写
- [ ] 用户故事粒度符合"单次迭代可完成"原则（过大则已拆分）
- [ ] 验收标准可量化、可验证
- [ ] 边界说明中有明确的 Not In Scope
- [ ] 文件已写入正确路径
- [ ] 已向用户展示摘要并请求确认
